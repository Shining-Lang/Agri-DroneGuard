# ADR-002：使用 Flyway 管理数据库迁移

## 状态

已采纳。

## 当前业务问题

打药申请、地块、审批和航线等数据结构会持续演进。开发环境、自动化测试和未来线上环境必须以相同顺序创建数据库结构，并且能够追踪每一次结构变化。

## 技术约束

- 主数据库是 PostgreSQL/PostGIS。
- Java 服务是业务数据结构的主要维护者。
- 迁移需要进入 Git，能够被审查和复现。
- 后续会包含空间字段、索引和 PostgreSQL 专用 SQL。

## 可选方案

- Hibernate `ddl-auto`：根据实体自动创建或更新表结构。
- Liquibase：使用 XML、YAML、JSON 或 SQL 描述变更集。
- Flyway：按版本顺序执行显式 SQL 迁移。

## 最终选择

使用 Flyway，并将版本化 SQL 放在 `backend/src/main/resources/db/migration`。生产代码不会使用 Hibernate `ddl-auto=update` 修改数据库结构。

## 选择理由

Flyway 的迁移模型简单，SQL 可以直接表达 PostGIS 类型、空间索引、约束和 PostgreSQL 特性。迁移文件进入 Git 后，数据库结构变化可以像代码一样被审查，并由应用启动流程自动校验。

## 替代方案的问题

Hibernate `ddl-auto` 适合原型，但自动推断的结构变化不够可控，难以审查高风险删除或类型修改。Liquibase 功能更全面，但变更集语法和维护成本更高；当前个人项目不需要复杂的跨数据库抽象。

## 极端场景表现

当多实例同时启动时，Flyway 使用数据库锁避免重复迁移。当迁移校验失败时，应用应拒绝启动，避免新代码运行在旧结构上。大表变更仍需拆分为兼容性迁移，Flyway 本身不能消除长时间锁表风险。

## 现实工程场景

### 场景一：多人分支产生重复迁移版本

- 背景：开发者 A 和开发者 B 都从主分支创建功能分支，当时最新迁移是 `V1`。
- 触发条件：两人分别创建 `V2__add_parcel.sql` 和 `V2__add_audit_log.sql`，代码合并后出现两个 `V2`。
- 系统表现：Flyway 在校验迁移时发现重复版本，应用启动或 CI 测试失败，不会猜测两个脚本的执行顺序。
- 处置方案：尚未进入共享环境时，为其中一个迁移重新分配版本；如果已有脚本执行到共享环境，则保留历史脚本并为另一项变更新建更高版本。
- 预防规则：当前单人开发可使用连续版本；进入多人协作后改用 UTC 时间戳版本，例如 `V20260628103015__add_audit_log.sql`，并在 CI 中执行 Flyway 校验。时间戳只能降低冲突概率，仍需代码审查确认版本唯一。
- 本项目落点：迁移统一存放在 `backend/src/main/resources/db/migration`，合并前必须运行后端集成测试。

### 场景二：已部署迁移被直接修改

- 背景：`V1__create_spray_applications.sql` 已经执行到测试或生产环境。
- 触发条件：开发者发现拼写或字段设计问题，直接修改原有 `V1` 文件并提交。
- 系统表现：文件校验和与 `flyway_schema_history` 中的记录不一致，Flyway 校验失败并阻止应用启动。
- 处置方案：恢复原迁移文件，然后新增更高版本的修复迁移，例如 `V2__fix_spray_application_schema.sql`。只有在确认数据库结构本身正确、所有环境都接受历史重写时，才可以经过审批使用 `flyway repair`；它不是常规修复手段。
- 预防规则：任何进入共享环境的版本化迁移都视为不可变；修正结构和数据只能追加新迁移。
- 本项目落点：Git 评审需要同时检查迁移版本、SQL 内容以及是否修改了历史文件。

### 场景三：大表结构变更阻塞线上请求

- 背景：未来审计日志或航线点表可能增长到数百万甚至更多记录，并存在长事务和持续写入。
- 触发条件：应用启动时直接执行需要重写数据或长时间持锁的 `ALTER TABLE`、普通索引创建或一次性全表回填。
- 系统表现：PostgreSQL DDL 等待锁或持有高等级锁，业务请求堆积、超时，滚动发布中的新实例也可能迟迟无法就绪。
- 处置方案：使用 expand-contract 流程：先增加兼容的可空字段，部署双写，分批回填，再切换读取，最后在后续版本收紧约束或删除旧字段。大索引使用 `CREATE INDEX CONCURRENTLY`，并合理设置 `lock_timeout` 与 `statement_timeout`。
- 预防规则：高风险迁移不得跟随普通应用启动直接执行；应在发布前评估锁、数据量、执行时间和回滚方案，并通过独立迁移任务执行。`gh-ost` 和 `pt-online-schema-change` 属于 MySQL 工具，不用于本项目的 PostgreSQL。
- 本项目落点：当前小表允许由 Spring Boot 启动 Flyway；达到生产规模后，将迁移从多实例启动流程拆为单独的发布步骤。

### 场景四：滚动发布期间新旧代码同时运行

- 背景：线上发布不会瞬间替换全部实例，新旧版本会在一段时间内同时服务请求。
- 触发条件：迁移立即删除旧列、重命名字段或新增强制非空字段，而旧版本代码仍依赖原结构。
- 系统表现：旧实例出现 SQL 异常，新实例可能写入旧实例无法识别的数据，导致部分请求失败。
- 处置方案：所有迁移先保持向后兼容；按照“扩展结构、迁移数据、切换代码、清理旧结构”的多次发布流程执行。
- 预防规则：数据库迁移和应用版本必须分别考虑前向兼容与后向兼容，破坏性清理至少延后一个稳定发布周期。
- 本项目落点：后续状态字段、审批表和航线表变更都采用追加式迁移，不在同一次发布中立即删除旧结构。

## 团队协作规范

- 迁移文件进入共享环境后禁止修改和删除。
- 一个迁移只处理一个可解释的结构或数据变化。
- 合并前必须检查版本唯一性，并运行 Testcontainers 集成测试。
- 数据删除、大表回填、非并发索引和字段类型重写必须单独评审。
- 生产迁移必须定义执行窗口、超时、监控指标、停止条件和恢复方案。
- `flyway repair` 必须经过人工确认，不用于掩盖普通校验失败。

## 代价与风险

- 已执行的迁移不能随意修改，只能新增修复迁移。
- PostgreSQL 专用 SQL 会降低跨数据库迁移能力。
- 复杂迁移需要人工设计回滚和数据修复方案。

## 项目落点

- 数据源配置：`backend/src/main/resources/application.yml`
- 迁移脚本：`backend/src/main/resources/db/migration`
- 迁移验证：后端 Testcontainers 集成测试

## 面试表达

我选择 Flyway 是因为项目后续包含 PostGIS 类型、空间索引和审批审计数据，需要显式、可审查、可复现的数据库变更。相比 Hibernate 自动建表，Flyway 在多环境一致性和失败阻断方面更可靠；相比 Liquibase，它更符合当前以 PostgreSQL SQL 为主的项目规模。

## 面试追问

### Q：两个开发者都创建了 `V2`，合并后会发生什么？

Flyway 会检测到重复版本并拒绝迁移，应用或 CI 会失败。未部署时重新编号即可；已部署的脚本不能改写，应保留历史并为另一项变更创建新版本。多人团队可以采用 UTC 时间戳版本并配合 CI 校验，但仍要通过评审保证唯一性。

### Q：为什么不能修改已经执行过的迁移？

Flyway 会记录迁移文件的校验和。修改历史文件会导致校验失败，这是为了防止代码声明的数据库历史与真实环境发生漂移。正确做法是恢复历史文件并增加修复迁移，而不是随意执行 `repair`。

### Q：Flyway 能否直接解决千万级大表在线变更？

不能。Flyway 管理迁移顺序和历史，但不会自动消除 PostgreSQL 的锁和数据重写成本。大表需要 expand-contract、分批回填、并发索引、超时控制和独立发布任务；Flyway 可以记录或编排这些步骤，但不能替代容量评估和发布设计。

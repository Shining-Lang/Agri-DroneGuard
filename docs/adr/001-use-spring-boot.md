# ADR-001：使用 Spring Boot 作为主业务后端

## 状态

已接受

## 背景

本系统需要一个可信业务后端，用来承载打药任务订单、地块归属校验、审批流程、审计日志、用户权限、Agent 规划结果持久化和模拟任务下发。

即使 Agent 服务失败、超时或者返回不合法结果，系统也必须保留业务状态，并能够继续恢复或重试。

## 决策

使用 Java 21 和 Spring Boot 3 作为主业务后端。

## 备选方案

- 使用 Python 做全栈后端。
- 使用 Node.js 做业务后端。
- 使用不依赖 Spring Boot 的轻量 Java 框架。

## 选择理由

Spring Boot 适合构建企业级业务系统，尤其适合处理事务、参数校验、权限、安全、分层架构、审计日志和长期维护。

本项目虽然包含 AI Agent，但它不是一个简单的聊天机器人，而是一个接近政企/农服业务流程的系统。因此主业务后端需要稳定、可控、可审计。

## 替代方案的问题

Python 非常适合 Agent、RAG、LangGraph 和算法工具，但如果把订单、审批、状态流转和审计全部放在 Python 主服务里，企业级后端表达会弱一些。

Node.js 开发效率高，但在本项目目标中，Java 后端更贴近求职方向，也更适合展示事务、状态机、权限和审批流。

轻量 Java 框架可以降低框架负担，但需要自行补齐很多 Spring Boot 已经成熟提供的能力。

## 极端场景表现

如果 Agent 在航线规划过程中崩溃，Java 后端仍然保存任务状态、审计记录和可重试的规划请求。

如果两个管理员同时审批同一个任务，Java 后端可以通过数据库事务、乐观锁或状态机约束保护状态流转。

如果 Agent 返回非法 JSON，Java 后端可以拒绝入库，并把任务保持在可恢复的规划失败状态。

## 代价与风险

Spring Boot 会带来一定框架复杂度和依赖管理成本。项目也需要清晰划分模块边界，避免后端服务逐渐变成职责混乱的大单体。

## 项目落点

`backend` 模块负责：

- 任务订单 API
- 地块与禁飞区 API
- 审批流程
- 审计日志持久化
- Redis Stream 任务投递
- Agent 规划结果接收
- 模拟飞控任务包生成

## 面试表达

我选择 Spring Boot 作为可信业务底座，因为这个项目不只是调用大模型，而是包含订单状态、审批流、审计日志、权限和 Agent 失败恢复。Python 负责 AI 规划，但 Java 负责可靠的业务边界。
